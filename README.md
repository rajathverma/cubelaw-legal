# cube-law-legal

Public legal pages for Cube Law Corporation, served at **terms.cubelaw.ca**.

This repo is the source of truth. GitHub Pages auto-publishes it.

---

## URL plan

The canonical URL never changes — quote emails point to it forever.
Each version is also frozen at a permanent dated URL.

| URL | Always shows | Type |
|---|---|---|
| `terms.cubelaw.ca/` | Landing page | Index |
| `terms.cubelaw.ca/engagement-terms` | Current Engagement Terms | **Canonical** |
| `terms.cubelaw.ca/engagement-terms-2026-04-28` | Frozen snapshot of v1.1 | Dated archive |
| `terms.cubelaw.ca/privacy` | Current Privacy Policy | **Canonical** |
| `terms.cubelaw.ca/privacy-2026-04-28` | Frozen snapshot of v2.0 | Dated archive |
| `terms.cubelaw.ca/privacy-2025-06` | Old June 2025 Privacy Policy | Archived |

When you publish a new version later, the existing dated URL stays put forever, and the canonical URL updates to the new version. **Quote emails that link to the canonical URL never need to change.**

## How the repo is organized

```
/
├── index.html                ← landing page at terms.cubelaw.ca
├── styles.css                ← shared stylesheet
├── _engagement-terms.html    ← SOURCE for current Engagement Terms (edit this)
├── _privacy.html             ← SOURCE for current Privacy Policy (edit this)
├── _privacy-2025-06.html     ← archived June 2025 Privacy (frozen, don't edit)
├── build.py                  ← generates the /out folder
├── README.md                 ← this file
└── out/                      ← generated by build.py — this is what gets published
    ├── index.html
    ├── styles.css
    ├── CNAME                 ← tells GitHub Pages to use terms.cubelaw.ca
    ├── engagement-terms/
    │   └── index.html        ← canonical URL
    ├── engagement-terms-2026-04-28/
    │   └── index.html        ← dated snapshot (with yellow archive banner)
    ├── privacy/
    │   └── index.html
    ├── privacy-2026-04-28/
    │   └── index.html
    └── privacy-2025-06/
        └── index.html
```

Files starting with `_` are sources you edit. The `/out/` folder is what gets deployed.

## One-time setup (~60 minutes total)

### 1. Create the public GitHub repo (5 min)

```bash
# At github.com, create new public repo: cubelaw/legal
git init
git add .
git commit -m "Initial: Engagement Terms v1.1, Privacy Policy v2.0"
git branch -M main
git remote add origin git@github.com:cubelaw/legal.git
git push -u origin main
```

### 2. Configure GitHub Pages (5 min)

1. GitHub repo → **Settings → Pages**
2. **Source:** Deploy from a branch
3. **Branch:** `main` / folder: `/out`
4. Save. GitHub publishes the `/out` folder at `https://cubelaw.github.io/legal/` within 60 seconds.

### 3. Connect the custom domain terms.cubelaw.ca (~30 min, mostly DNS)

**At GitHub:** Settings → Pages → Custom domain → enter `terms.cubelaw.ca` → Save. The CNAME file is already in `/out`.

**At your DNS provider** (where cubelaw.ca DNS is managed — could be GoDaddy, Namecheap, Cloudflare, your hosting provider, etc.):

Add this DNS record:
```
Type:   CNAME
Host:   terms
Target: cubelaw.github.io
TTL:    Auto / 3600
```

Wait for DNS to propagate (usually 5-30 min). Then back at GitHub Pages, tick **Enforce HTTPS** (wait ~10 min for the SSL cert to issue).

### 4. Verify (5 min)

Open these URLs and confirm they load:
- `https://terms.cubelaw.ca/`
- `https://terms.cubelaw.ca/engagement-terms`
- `https://terms.cubelaw.ca/engagement-terms-2026-04-28` (yellow banner appears at top)
- `https://terms.cubelaw.ca/privacy`
- `https://terms.cubelaw.ca/privacy-2026-04-28`
- `https://terms.cubelaw.ca/privacy-2025-06`

Test the mobile layout by opening one of these on your phone.

### 5. Set up WordPress 301 redirects (~10 min)

Goal: existing inbound links to `cubelaw.ca/privacy` and `cubelaw.ca/terms` should permanently redirect to `terms.cubelaw.ca`.

**Easiest method — Yoast SEO Premium** (or Rank Math, or Redirection plugin):
- Tools → Redirects → Add new
- From: `/privacy` → To: `https://terms.cubelaw.ca/privacy` → Type: 301
- From: `/terms` → To: `https://terms.cubelaw.ca/engagement-terms` → Type: 301

**Wait** — `cubelaw.ca/terms` is your **website Terms of Use** (different from Engagement Terms). Decide before redirecting:

- If you want the existing `cubelaw.ca/terms` page (Terms of Use) to keep working as-is, **don't redirect it**. Leave it.
- If you want all legal pages on terms.cubelaw.ca, you'll need a Website Terms of Use page there too — that's a separate piece of work.

**Recommendation:** redirect only `/privacy → terms.cubelaw.ca/privacy` for now. Leave `/terms` (Website Terms of Use) on cubelaw.ca for now and revisit later.

### 6. Update Deluge `button.SendQuoteEmail` (~5 min)

Add these constants at the top of the function:

```javascript
v_engagementTermsUrl = "https://terms.cubelaw.ca/engagement-terms";
v_privacyPolicyUrl = "https://terms.cubelaw.ca/privacy";
```

In the email template, append this paragraph just before the signature:

```html
<p style="font-size: 13px; color: #586E6C; margin-top: 16px;">
  This matter is governed by Cube Law Corporation's
  <a href="${v_engagementTermsUrl}" style="color: #4A7B6F;">General Terms of Engagement</a>
  and our
  <a href="${v_privacyPolicyUrl}" style="color: #4A7B6F;">Privacy Policy</a>.
  By submitting your intake form and retainer, you confirm you have read and agreed to both.
</p>
```

These URLs always serve the current version. You never have to update them again.

## Workflow when you update either policy

When you publish a new version (say, Engagement Terms v1.2 on 1 December 2026):

### 1. Edit the source

Edit `_engagement-terms.html`:
- Update the version stamps (hero block, footer)
- Make your content changes
- Update the "Earlier Versions" section in the related links to add v1.1 dated URL

### 2. Update build.py

Change the `DATE` constant from `"2026-04-28"` to `"2026-12-01"`.

### 3. Build

```bash
python3 build.py
```

This creates new files at:
- `/out/engagement-terms/index.html` (the new v1.2 — canonical URL updates)
- `/out/engagement-terms-2026-12-01/index.html` (frozen v1.2 — new dated URL)

### 4. Lock the previous version permanently

The previous canonical (v1.1, dated 2026-04-28) was already frozen at `/engagement-terms-2026-04-28/` when v1.1 published. That URL stays put — `build.py` doesn't touch existing dated folders, only writes the canonical and the *current* date. **You should never run `build.py` for a date that already exists in the repo or you'll overwrite the frozen version.**

If you're worried, verify by checking that `/out/engagement-terms-2026-04-28/index.html` is unchanged before committing.

### 5. Commit and push

```bash
git add .
git commit -m "Engagement Terms v1.2 — clarified section 7.3"
git tag -a engagement-terms-v1.2 -m "v1.2 effective $(date +%Y-%m-%d)"
git push && git push --tags
```

GitHub Pages auto-deploys within 60 seconds. The canonical URL now serves v1.2; the v1.1 dated URL still serves v1.1.

### 6. Notify clients on active engagements (if material)

Per Engagement Terms section 14.1, material changes need written notice to existing clients before they take effect.

## Why this design works

| Concern | How this solves it |
|---|---|
| WordPress plugin update wipes terms | These pages live on GitHub Pages, independent of WordPress |
| Need to prove what was in effect on date X | Each version is frozen at its dated URL forever |
| Quote emails break when terms change | Quote emails link to canonical URL only — never breaks |
| Need readable mobile experience | Built mobile-first (collapsible TOC, larger taps, tighter spacing) |
| Need an audit trail | Every edit is a Git commit with timestamp and signed history |
| Need a free, simple host | GitHub Pages — free, stable, automatic HTTPS |
